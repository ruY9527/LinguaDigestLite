## 一、核心目标
- 每次刷新时，只获取真正新增或修改的文章，避免重复下载和解析。
- 所有文章数据存储在 iOS 本地数据库中，做到读取快速、存储精简、自动清理。
- 优化用户感知的刷新速度和整体应用流畅度。

## 二、总体刷新流程
```
[用户触发刷新 / 后台定时刷新]
       ↓
检查本地最后更新时间及服务端标识（如存在）
       ↓
发起带“增量请求条件”的网络请求
       ↓
服务端响应：
  - 若无更新 → 仅更新本地“上次刷新成功时间”
  - 若有更新 → 下载并解析最新数据
       ↓
增量合并到本地数据库（去重、更新已存在条目）
       ↓
触发旧文章清理规则
       ↓
刷新界面列表（仅显示新增或变化的文章）
```

## 三、刷新触发逻辑优化
### 1. 手动刷新

- 用户下拉刷新或点击刷新按钮时，立即执行完整“条件请求”流程。
- 同时进行的多个手动刷新请求对同一订阅源只保留一个有效请求（请求去重）。

### 2. 后台自动刷新

- 为每个订阅源独立记录“上次刷新时间”。
- 系统根据用户使用习惯动态调整刷新频率：
    - 用户频繁阅读的源 → 较短刷新间隔（如 30 分钟）。
    - 长期未点击的源 → 较长刷新间隔（如 6 小时）。
- 利用 iOS 后台刷新机制，在系统资源允许时执行，避免耗电和流量浪费。

### 3. 首次启动或恢复订阅

- 首次添加订阅源时，拉取最近 N 条（如 20 条）文章，而不是全量历史。
- 若本地无任何数据，视为需要全量拉取，但仍尊重服务器分页（如 RSS 的 `max-items`）。

## 四、增量更新逻辑（核心优化）
### 1. 请求层面的增量标识

- 对每个订阅源，本地存储上次请求成功后服务器返回的 **时间戳** 或 **版本标识**。
- 下次请求时在 HTTP 请求头中携带该标识（如 `If-Modified-Since` 或 `If-None-Match`）。
- 若服务器返回“未修改”（状态码 304），则不再下载正文，仅刷新本地时间戳。

### 2. 解析层面的增量处理

- 下载的 feed 数据先解析出一批文章（每篇有唯一标识，如 `guid` 或 `link`）。
- 与本地已存在的文章标识进行比对：
    - **新标识** → 插入新记录，标记为“未读”。
    - **已存在标识** → 比较内容哈希或更新时间，若发生变化则更新字段（如标题、摘要、正文），否则跳过。
- 只对真正变化或新增的文章触发 UI 更新。

### 3. 分页与历史补全

- 对于支持分页的 RSS 接口（如 JSON Feed），优先只请求第一页（最新文章）。
- 当用户往下滚动加载更多时，再按需请求历史分页，并同样执行增量合并。

## 五、本地存储与去重逻辑
### 1. 文章唯一性约束

- 使用“订阅源 ID + 文章唯一标识（guid 或 permalink）”作为联合主键。
- 入库时自动去重：相同主键的文章不会重复存储。

### 2. 字段增量更新

- 对于已存在的文章，只更新可能变化的字段（如标题、内容摘要、发布时间），保留用户的本地状态（已读/未读、收藏、笔记）。

### 3. 订阅源元数据存储

- 每个订阅源记录：
    - 最后刷新成功时间
    - 上次使用的增量标识（时间戳或 ETag）
    - 当前总文章数（用于统计和清理）
    - 用户自定义的刷新优先级（高/中/低）

## 六、清理与过期策略
### 1. 基于时间的自动清理

- **已读文章**：保留最近 30 天，超过则删除。
- **未读文章**：保留最近 7 天，超过则自动标记为“已忽略”或删除（取决于产品定义）。
    - 用户可单独收藏的文章永久保留（不受时间清理影响）。

### 2. 存储空间压力触发清理

- 当本地数据库大小超过设定阈值（如 100 MB）时：
    - 优先删除最旧的已读文章。
    - 若仍超过阈值，则删除超过 14 天的未读文章。
- 清理后重建索引，确保查询性能。

### 3. 订阅源被删除时的处理

- 用户移除订阅源时，同时删除该源下的所有本地文章，释放存储空间。

## 七、并发与一致性
### 1. 多订阅源并发刷新

- 允许多个订阅源同时刷新，但限制最大并发数（如 3 个），避免网络拥塞和数据库锁竞争。
- 每个源的刷新在独立的后台任务中进行，互不阻塞。

### 2. 数据库写入事务

- 同一订阅源的多次网络响应（如分页）合并为一次数据库事务，避免碎片化写入。
- 批量插入或更新文章，而不是逐条操作。

### 3. 界面刷新与数据同步

- 刷新过程中，UI 先展示本地已有数据（不阻塞交互）。
- 增量数据到达后，仅插入/更新变化的部分，避免全表重载。
- 若刷新失败（网络错误或解析错误），保留原数据并提示用户，不回滚本地已有记录。
## 八、用户可感知的优化点
| 场景        | 优化前逻辑          | 优化后逻辑             |
| --------- | -------------- | ----------------- |
| 频繁下拉刷新同一源 | 每次都完整下载几十篇文章   | 命中 304，几乎无流量，瞬间完成 |
| 订阅上百个源    | 全量更新导致严重卡顿     | 按优先级并发，增量请求，界面流畅  |
| 文章被原站修改   | 本地会重复存为新文章     | 检测到标识相同，更新已有记录    |
| 存储过多旧文章   | 数据库查询慢，占用数百 MB | 自动清理，保持 <50 MB    |
| 离线或弱网刷新   | 多次超时失败         | 条件请求快速失败，不重复盲试    |
## 九、异常与边界情况处理
1. **服务器不支持增量标识**
    - 退化为全量拉取，但利用本地去重，只插入新的文章，不会重复存储。
2. **文章唯一标识变化**（如原站改了链接结构）
    - 用“标题 + 发布时间”组合模糊匹配识别可能是同一篇，但降低匹配严格度；用户可手动标记为重复。
3. **时间戳误差**（服务器时间不准）
    - 以服务器返回的 `pubDate` 为准，本地仅记录最后刷新时间，不做时间比较来判定更新。
4. **刷新中途应用退出**
    - 记录每篇文章的插入/更新状态，下次启动时继续未完成的事务（基于幂等设计）。
